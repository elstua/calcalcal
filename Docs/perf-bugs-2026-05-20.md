# Perf & Correctness Bugs ŌĆö 2026-05-20

Source: Backend server logs (Artem, dev environment) showing a single editor save
producing three GET /entries calls in 116ms, two PATCH cycles each spawning an
AI analyze job, and a streaks recalculation on every request.

This PRD covers three independent bugs surfaced by those logs. Bug 1 is the
expensive one (user data loss + token waste). Bugs 2 and 3 are perf/log noise
with no data loss.

Implement in order: Bug 1 ŌåÆ Bug 3 ŌåÆ Bug 2. Each ships as its own PR.

---

## Bug 1 ŌĆö iOS regenerates block UUIDs after every cache round-trip

### Severity
­¤ö┤ **HIGH.** Causes:
- User's edited calories to silently change back to AI estimates
- Token spend on every editor open (re-analyzes blocks the AI already processed)
- Backend `defensive dedup` warning spam (commit `388647a` covers the damage,
  but should not be the primary defense)

### Evidence
Backend log (single editor session, no content change between PATCHes):

```
20:54:41 PATCH entries/<id>      4 incoming vs 4 existing, "preserving nutrition"   ŌåÉ OK
20:56:22.323 PATCH entries/<id>  4 incoming vs 3 existing, "New block 107F98A1 added"
20:56:22.329 POST /api/ai/analyze  blocks=1                                          ŌåÉ AI fires
20:56:22.434 PATCH entries/<id>  4 incoming vs 1 existing, "New block 096E3227 added"
                                                            "New block 9C68CC9C added"
                                                            "New block D365C1C2 added"
```

The PATCH body claims "new" blocks but the user added nothing. iOS regenerated
three fresh UUIDs and sent them as additions.

### Root cause

iOS uses two identity fields per Block: `id: UUID` (the in-memory identity) and
`stableId: UUID?` (the cross-session identity that's supposed to round-trip with
the backend). The disk cache loses `stableId`.

Code paths:

1. **`calcalcal/Models/BlocksCache.swift` `encodeBlocks`** persists each block as
   `CachedBlock(id: block.id.uuidString, ...)`. There is no `stableId` field on
   `CachedBlock` at all.

2. **`calcalcal/Models/BlocksCache.swift` `decodeBlocks`** returns:
   ```swift
   Block(type: ..., ..., stableId: nil)   // line ~165
   ```
   ...and discards the persisted `id` (constructs a new Block without it).

3. **`calcalcal/Models/BlockModel.swift` `withStableIdsAndChangeTracking()`** is
   called by every consumer that hydrates a Block array (EntryCard line 88,
   DiaryTabView line 411, EditorOverlay line 249, DiaryAPI line 567). When
   `stableId == nil`, it assigns `UUID()` ŌĆö a fresh ID.

4. The next PATCH to `/api/diary/entries/<id>` ships those fresh UUIDs as block
   IDs. Backend sees them as new blocks. Inserts them, triggers AI analysis on
   their content, and replaces the user's typed nutrition with the AI's
   estimate.

The backend's defensive dedup in `DiaryEntry.updateContentAndBlocks`
(commit `388647a`) catches most of this by collapsing collisions by normalized
content. But that's a *secondary* defense ŌĆö it relies on the AI not changing
the text, and on `userModified: true` being correctly preserved through the
collapse. The primary fix is for iOS to send the same block ID twice for the
same block.

### Fix

Persist and restore `stableId` end-to-end through the disk cache.

#### `calcalcal/Models/BlocksCache.swift`

1. Add `stableId: String?` to `CachedBlock` (the on-disk Codable struct).
2. In `encodeBlocks`, set `stableId: block.stableId?.uuidString ?? block.id.uuidString`
   for every case in the switch (text / imageText / image / spacer).
3. In `decodeBlocks`, after constructing each `Block`, set:
   ```swift
   block.stableId = cb.stableId.flatMap(UUID.init(uuidString:))
                  ?? cb.id.flatMap(UUID.init(uuidString:))
   ```
   ...so legacy cache entries (where `stableId` is missing but `id` exists)
   still get a stable ID on first load.
4. Also restore `block.id` from `cb.id` so the in-memory identity matches what
   was saved. (Currently `decodeBlocks` creates Block without passing `id`,
   so SwiftUI sees a different identity than before ŌĆö minor source of
   re-rendering churn.)

#### Backward compatibility

`CachedBlock` is `Codable`. Adding `stableId: String?` as an optional field is
forward-compatible with existing on-disk caches (`JSONDecoder` defaults missing
optionals to nil). No version bump needed.

The fallback to `cb.id` in `decodeBlocks` ensures legacy cache entries (which
have `id` populated but no `stableId`) still produce stable IDs on first load.
From the next save onward, `stableId` is persisted explicitly and round-trips
cleanly.

#### Acceptance criteria

- After editor save and reopen, backend logs show `"Merged blocks=N, preserving
  nutrition data"` with **no** `"New block X added"` lines when the user did
  not add a block.
- No `POST /api/ai/analyze` job fires on reopen of an unchanged entry.
- `[DiaryEntry] Defensive dedup removed N duplicate block(s)` log message
  (currently fires on every save ŌĆö count in production via
  `docker logs calcalcal-api | grep "Defensive dedup" | wc -l` over a known
  window) drops to ~0.

#### Out of scope for this PR

- Migrating existing user data with already-drifted block IDs. New saves heal
  themselves. Old entries with duplicated blocks need the SQL fix from the
  `calcalcal` skill's "Fixing miscounted diary days" recipe applied
  case-by-case.

## Bug 2 ŌĆö iOS fires 3 GET /entries and 2 GET /entries/<id> per editor close

### Severity
­¤¤Ī **MEDIUM.** Wastes bandwidth, creates race conditions that the defensive
preserve-local-calories logic in `DiaryTabViewModel.apply(entries:)` exists
to paper over.

### Evidence
```
20:56:19.995  GET /api/diary/entries/6AE6CF61...
20:56:20.007  GET /api/diary/entries/6AE6CF61...  ŌåÉ duplicate, 12ms later
20:56:22.313  GET /api/diary/entries
20:56:22.318  GET /api/diary/entries               ŌåÉ duplicate, 5ms later
20:56:22.429  GET /api/diary/entries               ŌåÉ third in 116ms
```

### Root cause

Multiple `.onAppear` / `.onReceive(NotificationCenter.publisher(...))` /
`onChange` handlers each kick a `refreshVisibleEntries()` or similar after the
editor closes. From grep:

- `DiaryTabView.swift:78` ŌĆö onDismiss of all-days cover (with 500ms delay)
- `DiaryTabView.swift:116` ŌĆö `.onReceive(.diaryEntryCanonicalIdResolved)`
- `DiaryTabView.swift:122` ŌĆö `.onReceive(.streaksDataUpdated)`
- `DiaryTabView.swift:128` ŌĆö `.onReceive(.diaryEntryCaloriesUpdated)`
- `DiaryListView.swift:97, 110` ŌĆö same notifications
- `DiaryTabView.swift:110` ŌĆö `.onAppear`

After the editor closes, the iOS app posts at least 3 notifications
(`canonicalIdResolved`, `caloriesUpdated`, `streaksDataUpdated`), each
triggering a full refresh from at least 2 view-tree subscribers. There is no
debounce or in-flight-coalescing.

### Fix

Coalesce concurrent refreshes in `DiaryTabViewModel`:

1. Track a single `Task<Void, Never>?` for the current refresh.
2. `refreshVisibleEntries()` becomes:
   ```swift
   func refreshVisibleEntries() async {
       if let existing = currentRefreshTask {
           // Already in flight ŌĆö await the existing one instead of starting another.
           await existing.value
           return
       }
       let task = Task { @MainActor in
           await self.performRefresh()
           self.currentRefreshTask = nil
       }
       currentRefreshTask = task
       await task.value
   }
   ```
3. After the refresh completes, debounce subsequent calls within a short window
   (250ms is enough). Any caller arriving inside the window receives a no-op.

#### Acceptance criteria

- A single editor save produces **at most one** `GET /api/diary/entries` and
  at most one `GET /api/diary/entries/<id>` request, regardless of how many
  notifications fire.
- The existing `isLoadingRecentDays` guard in `refreshVisibleEntries` is
  superseded by the task coalescing (remove the early-return when this lands;
  it short-circuits silently and skips a legitimate refresh).

#### Risk

Coalescing the wrong refreshes. If a notification fires *during* a refresh
that started *before* the data change that triggered the notification, we
might miss the update. Mitigation: when coalescing, if the second caller would
have queried a wider date range than the first, run a follow-up refresh after
the first completes.

For PR scope, start with simple coalescing and observe in dev. If we
underrun, add a "dirty" flag that schedules one more refresh after the
current one finishes.

#### Out of scope

The bigger state-consolidation work (collapsing the NotificationCenter bus
into a single owner) is triage item #3 from the 2026-05-20 perf doc and is
deferred to a later PR.

## Bug 3 ŌĆö Backend recalculates streaks on every GET /api/streaks

### Severity
­¤¤Ī **MEDIUM.** Wastes CPU and DB queries on every navigation in the app.
Currently `GET /api/streaks` does a full `StreakCalculator` pass even when
the user has logged an entry today. Caching this is one of the easier wins.

### Evidence
```
[GET /api/streaks] Found streaks: current=1, total=12, lastEntry=Wed May 20 2026 00:00:00 GMT+0100 (Western European Summer Time)
[GET /api/streaks] Staleness check: lastEntryIsRecent=false, streakExpired=true, outOfSync=false
[GET /api/streaks] Recalculating streaks for user=... (expired=true, outOfSync=false)
```

`lastEntry` is May 20, today is May 20, but `lastEntryIsRecent=false`.

### Root cause

`apps/backend/node/src/routes/streaks.ts:30ŌĆō33`:

```ts
const today = new Date().toISOString().split('T')[0];        // "2026-05-20" (string)
const yesterday = new Date(Date.now() - 24*60*60*1000).toISOString().split('T')[0];
const lastEntryIsRecent = streaksData?.lastEntryDate &&
  (streaksData.lastEntryDate === today || streaksData.lastEntryDate === yesterday);
```

`streaksData.lastEntryDate` is a `Date` object (the JSON-stringified form in
logs is `"Wed May 20 2026 00:00:00 GMT+0100"`, i.e. the default JS Date
`toString()` representation), while `today` and `yesterday` are strings
(`"2026-05-20"`). Strict `===` is always false ŌåÆ `lastEntryIsRecent` is
always false ŌåÆ `streakExpired` is always true whenever the user has any
streak ŌåÆ recalculation runs on every request.

Two correctness issues compounded:

1. **Type mismatch** (the main one): comparing Date to YYYY-MM-DD string.
2. **Timezone mismatch** (latent): `today` is computed in UTC via
   `toISOString()`. For users in negative timezones near midnight,
   "today" in UTC is "tomorrow" locally. Logs already show
   `[StreakCalculator] Using timezone offset=0 minutes` ŌĆö server treats every
   user as UTC. CalCalCal users in Western European Summer Time, PT, EU
   timezones west of UTC will silently get yesterday's UTC date used as their
   "today".

### Fix

1. Normalize both sides of the comparison to YYYY-MM-DD strings before
   comparing:

   ```ts
   function toDayKey(d: Date | string | null | undefined): string | null {
     if (!d) return null;
     const date = typeof d === 'string' ? new Date(d) : d;
     if (Number.isNaN(date.getTime())) return null;
     return date.toISOString().split('T')[0];
   }

   const todayKey = toDayKey(new Date())!;
   const yesterdayKey = toDayKey(new Date(Date.now() - 24 * 60 * 60 * 1000))!;
   const lastEntryKey = toDayKey(streaksData?.lastEntryDate);

   const lastEntryIsRecent =
     lastEntryKey !== null &&
     (lastEntryKey === todayKey || lastEntryKey === yesterdayKey);
   ```

2. (Same PR or follow-up ŌĆö your call) Read the client's timezone offset from
   the request and compute `today` / `yesterday` in that timezone, the same
   way `StreakCalculator` already does internally. Mobile apps already send
   `X-Timezone-Offset` style headers ŌĆö check `apps/backend/node/src/middleware`
   for an existing helper; the StreakCalculator path uses
   `req.query.timezoneOffset` per `streaks.ts:159`. Match that.

#### Acceptance criteria

- Logging an entry on day D and then opening the app on day D again must
  print `lastEntryIsRecent=true, streakExpired=false, outOfSync=false` and
  must NOT call `StreakCalculator.recalculateAllStreaks(userId)`.
- Logging an entry yesterday and not today (current streak 1, lastEntry
  yesterday) still prints `lastEntryIsRecent=true` until the day rolls over.
- Manual `?recalculate=true` query param still forces recalculation
  (existing behavior preserved).

#### Risk

Streak data going stale if the staleness check is too permissive. Mitigation:
keep the `outOfSync` branch ŌĆö it queries `DiaryEntryModel.listAnalyzedByDateRange`
and will still trigger recalc if the DB has analyzed entries the streaks row
doesn't reflect.

#### Out of scope

`StreakCalculator` itself (it works correctly given the right inputs and
already debounces). The 1-second debounce in
`Streak update scheduled for user=... (1000ms debounce)` is fine and stays.

## Suggested PR order and naming

1. `fix(ios): persist stableId in BlocksCache to stop block-UUID drift` ŌĆö
   addresses Bug 1. Highest priority. Smallest blast radius (one file:
   BlocksCache.swift).

2. `fix(backend): compare streak lastEntryDate as day-key strings` ŌĆö
   addresses Bug 3. One file (`routes/streaks.ts`). Trivial. Drops backend
   CPU on every navigation.

3. `perf(ios): coalesce concurrent refreshVisibleEntries calls` ŌĆö
   addresses Bug 2. One file (`DiaryTabViewModel.swift`). Riskier; ship after
   1 and 3 have soaked in dev.
